# Listener Retry and Backfill Model

## Scope

This document defines the bounded retry and backfill model for ORMB's mock USDT deposit listener. ORMB remains a testnet-only, mock-asset-only demo. This is not production stablecoin infrastructure and must not process real USDT, real RMB/CNH, real customer funds, or mainnet activity.

## Deterministic Backfill Planning

`workers/listener-backfill.ts` provides `createBackfillPlan`, a pure helper that splits an inclusive block range into bounded batches.

The planner fails closed when:

- block numbers are negative
- `fromBlock` is greater than `toBlock`
- `batchSize` is zero or negative
- `maxBlocks` is zero or negative
- the requested inclusive span exceeds `maxBlocks`

This prevents accidental unbounded historical scans in local or hosted-demo environments.

## Retry Policy

`decideRetryDelay` provides deterministic capped exponential backoff:

- attempts are 1-indexed
- retry delay starts at `baseDelayMs`
- delay doubles each attempt
- delay is capped at `maxDelayMs`
- retry stops when `attempt >= maxAttempts`

The helper does not sleep or perform network I/O. Future worker runners can use it to schedule safe retries around RPC reads, database writes, or bounded backfill batches.

## Safe Backfill Procedure

Future backfill tooling should:

1. Run in dry-run mode first.
2. Require explicit chain ID and token/treasury addresses.
3. Use a bounded block range generated by `createBackfillPlan`.
4. Use `(chainId, txHash, logIndex)` as the idempotency key.
5. Compare stored and observed block hashes before accepting an existing event.
6. Write audit logs for new, duplicate, ignored, and reorged events.
7. Avoid mint request creation until deposit reconciliation passes.
